HOW SHOULD WE BUILD ALL 6 FEATURES?

A
UI-FIRST, THEN MECHANICS
Ship the 3 pure UI features first (ticker, buffs bar, resonance bar, rich tooltips) as one pass — no state changes needed, low risk, immediately visible. Then add the 2 mechanics (void leeches, null fracture) in a second pass that wires in new state/reducer logic.
Pass 1: Ticker + Buffs Bar + Resonance Bar + Tooltips
Pass 2: Void Leeches + Null Fracture
  • Fastest visible results
  • Low regression risk
  • Two separate passes
  • Slight context switching
B
SHARED INFRASTRUCTURE, ONE PASS RECOMMENDED
Start by extending GameState with shared fields (activeBuffs[], leeches[], etc.), then implement all 6 features together. All state changes land once, all components are written fresh. Slightly more upfront planning but no half-baked state during dev.
Step 1: Extend types + reducer
Step 2: Ticker + Resonance Bar
Step 3: Buffs Bar + Tooltips
Step 4: Void Leeches + Null Fracture
  • Clean single state migration
  • Shared infrastructure (buffs used by leeches + fracture)
  • No backwards-compat hacks
  • Requires full spec upfront
C
FEATURE-BY-FEATURE, SHIP EACH IMMEDIATELY
Implement and verify each feature one at a time: ticker → resonance bar → buffs bar → tooltips → leeches → null fracture. Each feature is runnable and tested before the next starts. Slowest but safest, with live feedback at each step.
1: Ticker
2: Resonance Bar
3: Buffs Bar
4: Tooltips
5: Leeches
6: Null Fracture
  • Play-test after each
  • Easiest to catch issues
  • State touched multiple times
  • Leeches/fracture may need buffs-bar state to already exist
